# 詳細設計

## インターフェース

- `specflow.py generate/check/apply`: 要件正本を読取り、日本語派生文書を生成・比較し、版付き変更セットを原子的に適用する。
- `standardsflow.py generate/check`: 出典台帳を検証し、日本語の出典一覧を生成・比較する。
- `update_checklist.py`: 固定参照SWEBOKのSHA-256を確認し、既存ワークブックを正規化・拡張して`checklist.xlsx`を置換する。
- `devflow.py catalog/set-checks/inspect/advance/audit`: チェック項目カタログと案件別実施結果を管理し、工程ゲートを判定する。
- `validate_repo.py`: Skills、agents、配布マニフェスト、日本語文書、正本派生物を横断検査する。

CLIは既存サブコマンドと引数を維持する。実施結果スキーマ2の追加属性はJSONオブジェクトの後方互換な追加とし、既存スキーマ1の読取を継続する。

## データモデル

- 要件正本: `catalog_revision`と安定IDを持つ原子的要件。廃止は削除せず状態で保持する。
- 出典正本: Source ID、正式名称、版・公開日、URL、参照日、適用範囲、変更確認日、前版との差分、固定参照物、SHA-256を持つ。
- チェック項目: ID、工程、プロファイル、基準重要度、一つの統制、一つの合格基準、意図、出典、重複統制群を持つ。
- 実施結果スキーマ2: 適用性、判定、案件重要度と根拠、証跡、Issue、対応方針、期限、レビュアー、レビュー日、再確認、履歴を持つ。
- ワークブック実施列: `案件重要度`から`再確認結果`まで15列を各レビューシートへ付加し、マスターと案件実績を同一行で対応付ける。

## 制御・状態・例外

品質ゲートは`undecided → applicable/not-applicable`、適用時は`unreviewed → pass/fail`で遷移する。`not-applicable`にも具体理由、レビュアー、日付を要求する。Passは到達可能な直接証跡、FailはIssue・対応方針・未来または実行時点で有効な期限を必須とする。FailからPassへの変更は再確認情報を要求し、旧Failを履歴へ保存する。案件Criticalの未解決Fail、必須記録欠落、証跡不達はゲートを閉じる。

正本・ワークブックは候補ファイルを全検証してから`os.replace`で置換する。版競合、添付ハッシュ不一致、未知の出典ID、重複ID、全行数式、生成差分は例外終了し、旧成果物を維持する。

## 認証・認可・入力検証

常駐APIや利用者認証は追加しない。初回承認は要件・計画文書のダイジェストと要求者の記録で拘束する。以後の工程に追加承認は設けず、承認済み範囲外の変更だけを停止条件にする。JSONは型・列挙値・必須属性・安定ID・版を検証し、ファイル証跡はリポジトリ境界内の到達可能性を検査する。外部添付は実行せず、形式とハッシュだけを入力として扱う。

## 観測可能性

各ゲート判定を`reports/latest-<phase>.json`と時刻付きレポートへ出力する。状態遷移と結果更新は`events.jsonl`へ連鎖ハッシュ付きで記録する。失敗時は工程、項目ID、欠落属性、期待条件を機械可読なblockerとして返す。ワークブックはサマリ数式、出典マスター、KA対応表、追加項目管理で追跡可能性を可視化する。

## 移行・互換性

既存実施結果スキーマ1は読取・監査を継続し、新規work itemだけをスキーマ2で作成する。既存の承認済みwork itemはダイジェストを変える自動移行を行わない。旧`SWEBOK-V4`参照は正本変更セットで`SWEBOK-V4A`へ更新し、ワークブック内の`swebok-v4.pdf`表記を生成時に正規化する。添付PDFは`.workspace/swebok-v4a.pdf`へ置くが配布物に含めない。
