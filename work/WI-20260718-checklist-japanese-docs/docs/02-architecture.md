# アーキテクチャ

## コンテキストと境界

本リポジトリは、他リポジトリへコピーして利用するSkillsと開発ガバナンスの参照実装である。今回の変更は外部サービスを追加せず、次の正本と派生物の境界を明確にする。

- 永続要件の正本: `spec/requirements/requirements.json`
- 公式資料の正本: `governance/standards/registry.json`
- チェック項目の生成元: `update_checklist.py`と固定参照SWEBOKの版・ハッシュ
- 実施結果の正本: `work/<ID>/review/checklist-results.json`
- 人間向け派生物: `docs/requirements/REQUIREMENTS.md`、`docs/standards/SOURCES.md`、`checklist.xlsx`、チェック項目カタログ

添付PDFは生成時のローカル入力に限定し、Git管理境界の外に置く。GitHubは完成したソース・派生物・検証結果をPull Requestで共有する公開境界とする。

## コンポーネントと責務

| コンポーネント | 責務 | 所有データ | 依存先 | SLO |
|---|---|---|---|---|
| 要件フロー | 要件正本の競合検査、原子的変更、派生文書生成 | 要件ID、revision、受入基準、trace | JSON Schema、Git | ローカル処理のためSLOなし |
| 出典フロー | 公式URL、版、範囲、差分、鮮度、添付ハッシュの検証 | 出典台帳 | 公式一次情報、ローカル添付 | ローカル処理のためSLOなし |
| チェックリスト生成 | チェック項目、プロファイル、原子性、集計式、実施列の生成 | `checklist.xlsx`、catalog | 出典台帳、添付SWEBOK | 決定的再生成 |
| 品質ゲート | 適用性、合否、証跡、是正、再確認、履歴の検証 | work itemの実施結果 | checklist catalog、Git作業ツリー | 欠落時は閉じる側へ失敗 |
| 文書生成 | 正本から日本語の利用者向け文書を生成 | Markdown派生物 | 要件・出典正本 | バイト単位の差分なし |

## データフローと信頼境界

```mermaid
flowchart LR
    U[ユーザー要求] --> W[work item・初回承認]
    E[外部添付・公式Web] -->|形式・版・ハッシュを検証| L[ローカル作業ツリー]
    W --> L
    L --> C[正本JSON]
    C --> G[決定的な生成器]
    G --> D[日本語文書・カタログ・Excel]
    D --> Q[品質ゲート・監査]
    Q -->|全検査成功時だけ| H[GitHub Pull Request]
```

図中の矢印は情報の流れ、外部添付・公式Web、ローカル作業ツリー、GitHub Pull Requestの境界は信頼レベルが変わる箇所を示す。

1. ユーザー要求をwork itemへ記録し、要件差分と実行計画を初回承認へ固定する。
2. 添付SWEBOKの版・ページ数・SHA-256を検査し、正本台帳にはメタデータだけを保存する。
3. 正本JSONを厳格検証してから原子的に置換し、日本語文書を再生成する。
4. チェックリスト生成器が正本・固定参照情報からワークブックと機械可読カタログを生成する。
5. 品質ゲートが各工程の実施結果と到達可能な証跡を検証し、未解決Failを後工程へ通さない。
6. ローカル検証成功後だけGitHubへブランチとPull Requestを公開する。

信頼境界は、外部添付・公式Web、ローカル作業ツリー、GitHubの3つである。外部入力は実行せず、PDF/Excelとして構造とハッシュだけを読み取る。

## 可用性・性能・拡張性

常駐サービスではないため可用性やマルチAZは非該当。性能上の主な制約はワークブック数式であり、Excelの全行参照を禁止して実データ最終行までに限定する。項目数の増加は生成時に範囲を更新し、プロファイルで実行対象を絞る。正本と生成器をコピーすれば、対象リポジトリで追加サービスなしに利用できる。

## 代替案とトレードオフ

- 添付PDFをコミットする案は再現性が高いが、再配布権が明確でないため不採用。正式名称、版、URL、ハッシュで同一性を担保する。
- 1,740項目の基準重要度を一括で下げる案は識別力を見かけ上改善するが根拠を失うため不採用。案件重要度と根拠を必須にし、案件Criticalだけを停止条件にする。
- 全チェックを無条件適用する案は重複・過剰統制になるため不採用。Core、Cloud Common、ベンダー差分、条件付きプロファイルを合成する。
- 英語正本のまま表示だけ翻訳する案は保守者の認知負荷が残るため、識別子・正規化actionを除く人間向け正本文言も日本語化する。

## 失敗・縮退・復旧

スキーマ不一致、古い版、添付ハッシュ不一致、全列数式、生成差分、到達不能証跡、根拠なしN/A、未是正Fail、再確認欠落は処理を失敗させる。正本更新は候補全体の検証後に原子的置換するため、途中失敗時は旧版を維持する。派生物は正本から再生成でき、Git差分とrevisionで直前状態へ追跡できる。
