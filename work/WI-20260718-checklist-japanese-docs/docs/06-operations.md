# 運用設計・運用引継ぎ

## SLO・SLI・エラーバジェット

常駐サービスを提供しないため稼働率SLOはN/A。品質SLIとして、正本派生差分0、カタログ差分0、監査blocker 0、単体テスト失敗0、ワークブック数式エラー0を採用する。mainへの変更はこれらをGitHub Actionsで満たすことを必須とする。

## 監視・ログ・アラート

GitHub ActionsのGovernance jobを変更時監視とし、失敗したコマンドとテスト名をログへ出す。work itemでは工程レポート、`events.jsonl`の連鎖ハッシュ、実施結果の履歴を保持する。出典は`change_checked_at`と`refresh_days`で期限超過を検出する。

## Runbook・インシデント対応

生成差分は正本を確認して再生成し、正本を派生物へ合わせて手編集しない。添付ハッシュ不一致は処理を止め、公式入手元と版を再確認する。未解決FailはIssue、対応方針、期限を登録し、修正後に別レビュー情報でPassへ再確認する。CI失敗は同じ承認範囲内で原因を修正し、全検査を再実行する。

## バックアップ・復旧・DR

Git履歴を正本・生成器・work記録のバックアップとする。派生Markdown、カタログ、ワークブックは正本と固定参照メタデータから再生成する。常駐データストア、RTO/RPO、マルチAZ/リージョンDRは非該当。固定SWEBOK本体は利用者が公式入手し、SHA-256で同一性を確認する。

## 容量・コスト・依存先

主な容量要因は1,740項目のワークブックとJSONカタログである。数式を実行範囲へ限定し、未使用セルへ書式・式を展開しない。CIはPython 3.12と固定開発依存だけを使用する。外部の継続依存はGitHub Actionsと出典確認時の公式Webで、実行時クラウド費用はない。
