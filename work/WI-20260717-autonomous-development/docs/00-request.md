## ユーザー原文

https://github.com/tsuji-tomonori/dev-standard 都度承認を求めるのはAI開発と相性が悪いため、最初の要件だけ承認を求め、以降は最後まで自走して。このとき、最初の要件決めと実行計画を立て、実行計画時にどのような作業が必要となるかを洗い出し、以降一切の承認なく最後まで走り切れるようにして。また、以降の対応も行って。skillsを作りたいです。過度な共感はしないが適切に傾聴し、言語化ができていないあいまいな部分は、意図をくみ取ったうえで言語化を行いながら、対話形式であいまいな部分は確認を行い、かといって出来ていない部分やあいまいな箇所に対して正論で確認するわけではなく、融和的に、かといって嫌味にならずに確認を行う、そんなskillsが欲しい。

言語化能力を高め、情報量は落とさず、シンプルにかつ文章量を抑えるための方法、核心を突く、意図をくみ取ることをより具体的にskillsに残して。

## 目的・期待する成果

開発開始前に、要件、実行計画、許可する操作、完了条件を一度だけ人が承認する。承認後はAIエージェントが品質ゲートを維持したまま、実装、検証、GitHubへの反映、完了記録まで自律継続する。加えて、曖昧な意図を融和的かつ簡潔に言語化するrepo-local skillを追加する。

## 対象範囲

- `dev-standard` の工程承認モデル、決定論的ハーネス、テスト、文書、フック、repo-local skillsを整合させる。
- 要件工程に実行計画を必須成果物として追加し、その単一承認を後続作業の権限境界にする。
- 承認後は後続工程で人の承認を要求せず、検査、証跡、トレーサビリティ、監査を維持する。
- `calibrated-collaborative-listening` をrepo-local skillとして追加し、添付調査の中核規則、具体的な言語化手順、評価基準を組み込む。
- feature branch、コミット、PR、CI確認、条件を満たしたPRのsquash mergeまでを実施する。

## 対象外

- SWEBOKチェックリストの合格基準自体を緩和すること。
- AIが人名義の承認、リスク受容、法的判断を偽装すること。
- 当初の権限境界にない課金、秘密情報の公開、データ削除、本番変更、外部連絡を行うこと。
- 臨床的カウンセリング、診断、心理治療をskillの目的に含めること。
- 過去に作成済みのwork itemを無条件に新方式へ読み替えること。

## 制約・前提

- 初回承認は、要件、トレーサビリティ、実行計画、外部変更範囲の同一ダイジェストへ結び付ける。
- 承認後の要件・実行計画は凍結する。変更が必要な場合、現在のwork itemを勝手に拡張せず、新しい要求として分離する。
- 低リスクかつ可逆な不確実性は、前提を記録して自走する。
- 後続工程の品質検査は承認ではないため、省略しない。
- 計画内のGitHub操作は `tsuji-tomonori/dev-standard` のfeature branchとPRに限定する。
- PRは全必須検証成功後にsquash mergeする。必須の外部レビューや権限不足で機械的に完了できない場合だけ、ブロッカーとして終了報告する。
