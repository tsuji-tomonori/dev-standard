# 要件定義

## 文書情報

- 版: 1.0-approval-candidate
- 基準日: 2026-07-17
- 目的: 単一の初回承認で安全に自走できる開発フローと、校正された協働的傾聴skillの要求を定義する。
- スコープ: `tsuji-tomonori/dev-standard` の統制フロー、ハーネス、文書、テスト、repo-local skill。
- 対象読者: 要求者、AI開発エージェント、実装・レビュー・監査担当者。

- 要求語: 「必須」は受入条件を満たさなければ完了不可、「推奨」は合理的理由がある場合に限り代替可能、「許可」は選択可能を表す。

## 用語

| 用語 | 定義 |
|---|---|
| 初回承認 | 要件工程で一度だけ行う、人による明示的な実行許可。要件と実行計画のダイジェストへ結び付く。 |
| 権限境界 | 初回承認で許可された対象、操作、外部副作用、禁止事項、完了条件の集合。 |
| 品質ゲート | 文書、チェック結果、証跡、テスト、監査を決定論的に検査する仕組み。人の承認とは区別する。 |
| 自走 | 権限境界内で、追加の承認要求を行わず、次の未完了作業を選び続けて完了まで進むこと。 |
| consequential ambiguity | 回答によりスコープ、危険性、費用、外部副作用、不可逆性が変わる曖昧さ。 |
| 核心 | 話題名ではなく、目的、障害、守る価値、次の判断点を最小の命題にしたもの。 |
| 意味保存圧縮 | 重複表現を削りつつ、要求、禁止、数値、否定、条件、例外、確度、帰属を保持する短文化。 |

## ステークホルダー

| 役割 | 関心事 | 責任 |
|---|---|---|
| 要求者 tsuji-tomonori | 都度承認を排し、品質を落とさず完了まで自走すること | 初回要件・実行計画を一度だけ承認する |
| AI開発エージェント | 権限境界と品質ゲートが機械的に判定可能であること | 計画、実装、検証、証跡、公開、完了を継続する |
| 将来のリポジトリ利用者 | フローが理解可能で、意図しない外部操作が起きないこと | 文書とコマンドに従い利用する |
| セキュリティ・監査担当 | 承認の真正性、証跡、改ざん検知、権限逸脱防止 | 独立レビューを行える。工程承認は要求しない |
| skill利用者 | 詰問・迎合・冗長さなしに意図が言語化されること | 仮説を短く修正できる |

## 優先順位規則

- P0: 自走と権限境界、安全性、承認真正性、品質ゲートの根幹。未達ならマージ不可。
- P1: 傾聴・言語化品質、文書整合、互換性、運用性。未達なら原則マージ不可。
- P2: 補助的な表現改善。P0/P1へ影響しない範囲で延期可能。

## データ・法令・倫理要件

- 個人データ、顧客データ、認証情報、本番ログを新規収集しない。
- 添付研究は設計根拠として要約し、論文本文を再配布しない。参照はタイトルと公開URLに限定する。
- 傾聴skillは日常的な対話支援であり、臨床診断や治療を標榜しない。
- AIの意図推測は仮説として校正し、感情、動機、診断、道徳判断を事実として断定しない。
- 初回承認の人物同一性をAIが捏造しない。要求者がこの会話で明示した判断だけを記録する。

## リスクとトレードオフ

| ID | リスク | 対応 |
|---|---|---|
| RSK-001 | 初回計画の漏れにより後工程で判断が必要になる | 実行計画に対象、操作、検証、外部副作用、rollback、停止条件を列挙する。低リスク事項には既定値を定める。 |
| RSK-002 | 承認削減が品質ゲート削減へ誤読される | 人の承認と決定論的品質検査を用語・実装・テストで分離する。 |
| RSK-003 | 承認後に要件を変更して権限を拡張する | 初回承認を要件・計画ダイジェストへ結び付け、全後続advanceとauditで再検査する。 |
| RSK-004 | 自走が権限外の外部操作へ拡張される | 権限境界と禁止事項を計画に固定し、範囲外は実行せずブロッカーとして終了する。 |
| RSK-005 | 傾聴が迎合、診断、冗長な共感へ崩れる | 事実・感情・判断・行動を分離し、critical failure付きrubricで検証する。 |
| RSK-006 | 短文化で条件や例外が消える | 原子的意味台帳と意味チェックサムをskillに含め、保護対象の欠落をcritical failureにする。 |
| RSK-007 | 既存work itemが新方式で誤って進む | schema/versionを検査し、明示移行されない旧状態は自動認可しない。 |

## 対象外・N/A判断

- 本変更はローカルCLI、文書、repo-local skillの変更であり、常時稼働サービス、UI、ネットワークAPI、データベース、クラウド資源を追加しない。
- 性能、容量、可用性、災害復旧のサービス水準は非該当。代わりにテスト再現性、監査性、保守性を測定する。
- 学習、推論、RAG、ファインチューニング、モデル配備は行わない。AIプロファイルはagent authorityと責任ある対話設計の観点に限定する。
- 人命、重大インフラ、高額取引を扱わないため、形式手法は必須としない。
- ユースケース図やUIプロトタイプは、分岐の少ないCLI統制変更には使用しない。状態遷移はFLOW文書とテストで検証する。

## 要求管理

- 正本はこのwork itemの `docs/01-requirements.md` と `docs/01-execution-plan.md` とする。
- 初回承認後は両文書を凍結する。実装上の詳細は設計・決定記録へ追加できるが、要求または権限境界を拡張してはならない。
- 権限境界外の新要求は別work itemへ分離する。現在のwork itemを無断変更しない。
- 要求状態は `承認候補 → 承認済み → 実装済み → 検証済み` とする。
- 変更率は初回承認後の要求差分件数で測定し、本work itemの許容値を0件とする。

- すべて本work item内で完了させる。延期項目は設けない。
