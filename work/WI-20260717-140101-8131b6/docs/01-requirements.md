# 要件定義

## 概要

「敵対的」を、対象は誤っているかもしれないという反証姿勢で行う独立・批判的レビューとして定義する。

## ステークホルダー

| 役割 | 関心事 | 承認責任 |
|---|---|---|
| requester | 意図どおりの非セキュリティ批判レビュー | 初回方針決定 |
| Skill利用者 | 短い指示で一貫した欠陥探索 | なし |

## 機能要件

| ID | 要件 | 優先度 | 受入基準 |
|---|---|---|---|
| REQ-001 | 前提として誤りの可能性を置き、反例・矛盾・欠落・未証明仮定を探す | Must | SKILL.mdに明記 |
| REQ-002 | 要件・設計・実装・テスト・文書を対象にする | Must | challenge playbookに全対象を掲載 |
| REQ-003 | security red teaming固有の脅威モデル・攻撃手順を中核から除く | Must | 旧語彙の回帰検査が成功 |
| REQ-004 | 独立再導出、複数視点、反証可能な主張、反例、mutation/metamorphic testingを選択可能にする | Must | Skillとresearch mapに対応規則あり |
| REQ-005 | 指摘を違反主張・最小証拠・影響・確信度・修正確認として報告する | Must | report templateに項目あり |
| REQ-006 | Skill一覧・配布・chat-first連携を新名称と意味へ同期する | Must | validatorとtestsが完全一致を検証 |

## 非機能要件

| ID | 品質特性 | 測定方法 | 合格閾値 |
|---|---|---|---|
| NFR-001 | 簡潔性 | quick_validateと行数確認 | Skill 500行未満、validation成功 |
| NFR-002 | 意味整合性 | rgと契約テスト | security攻撃を主目的とする語彙なし |
| NFR-003 | 移植性 | install tests | folder copyのみで使用可能 |

## データ・法令・倫理要件

実データや外部サービスを扱わない。人への攻撃・嫌味・人格批判を禁止する。

## リスクとトレードオフ

名称変更により既存参照の互換性は失われるが、PRは未マージのため曖昧な名称を残すより明確化を優先する。

## 対象外・N/A判断

security red teaming、侵入試験、運用SLO、データ移行は対象外。
