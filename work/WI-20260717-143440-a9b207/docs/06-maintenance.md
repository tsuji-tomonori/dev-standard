# 保守・変更・廃止計画

## 所有者と保守体制

Repository maintainerがskills/scripts/schema/profileを所有し、requesterがproduct requirement deltaを決定する。AIはissue/PR作成と検証を実行できるが、要件変更の初回承認を捏造しない。変更はbranch PR、全CI、reviewを通す。

## 依存関係・脆弱性・更新方針

Python 3.12、openpyxl 3.1.5、PyYAML 6.0.2、SQLGlot 27.28.1、Ruff 0.12.4をpinする。Dependency更新はrelease note/CVE/API差分を調査し、SQL/CFn/generated byte/installer regressionを全実行する。公式standard sourceはregistry期限前にissuer pageを再確認する。CI action major version更新もPR review対象とする。

## 互換性・移行方針

Canonical schemaは`schema_version`、catalogとitemは`revision`で管理する。Add/update/retire operationはexpected revisionを要求し、last-write-winsを許さない。Skill folder/profile pathを安定interfaceとし、target固有の`AGENTS.md`/`.codex/config.toml`を上書きしない。Breaking schema/profile changeはmigration、old/new fixture、release noteを伴う新しいauthorized deltaにする。

## EOL・データ廃棄

要件はphysical deleteせずretired tombstoneとして理由とrevisionを保持する。Skillを廃止する場合はcatalog entry、distribution profile、docs、testを同一PRで更新し、移行先を明示する。Personal/production dataは保存しないためdata disposal jobは非適用。Repository廃止時はhosting側retention policyとGit archiveでsource historyを扱う。
