# 振り返り・改善

## 成果と計画差

計画した3本柱、自己適用、移植profile、CI統合を実装した。ScopeはFastAPI/CDKのまま維持し、Leanや追加frameworkは導入していない。計画差は、adversarial reviewでrouter decorator混入、nested call順、`functions.py` digest漏れを発見し、設計契約とtestを強化したことである。

## 有効だった統制

Canonical/item二重revision、complete-candidate validation、generated byte check、official-host/freshness registry、temporary target install、negative fixturesが早期に具体defectを示した。Requirements > implementation > generated designのauthority順が、実装一致と要件適合を混同しない判断に有効だった。

## 手戻り・ゲート失敗・見逃し

最初のstandards host allowlistが`docs.aws.amazon.com`を許可せず、official AWS URLを拒否した。修正してamazon.com issuer hostを明示し、registry asset/canonical equality testを追加した。Venv symlinkがexecution environmentのpath変換で次callから利用不能になったため、repository-local site-packagesをsystem Pythonの`PYTHONPATH`から参照して検証を継続した。Product code/CI契約には影響しない。

## 根本原因

AWS document hostをサービスAPI hostの`amazonaws.com`と同一視したこと、AST source orderをruntime evaluation orderとみなしかけたこと、sequence sourceをrouterだけに狭く考え具体処理fileのdriftを見落としたことが原因である。いずれもpositive fixtureだけでは検出しにくく、independent negative oracleが必要だった。
